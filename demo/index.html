Translation Engine

Schema translation for heterogeneous tracking systems used in design and cross-functional work. Every pass produces a manifest naming what was preserved, what was lost, and how.

Slice 1 — Jira → Airtable, project management context

The problem

Most teams run multiple trackers — Jira for engineering, Airtable for roadmap, Notion for OKRs, Figma comments for design critique. Keeping work in sync between them typically results in some kind of loss in data quality. Priorities mean different things in different systems. Hierarchies flatten. Custom fields evaporate. Context lives in one place and is invisible in the next.

Most attempts to bridge these systems either pretend the loss isn't happening or hide it behind one-off engineering work that papers over the gap. Translation Engine takes a different approach: it names the loss explicitly, the same way every time, and ships that naming as the deliverable. The destination write is secondary; the artifact is a record of what survived translation, what didn't, and why.

Five kinds of loss

Every translation decision lands in one of five buckets. The taxonomy is the doctrine — the same five kinds apply whether the pair is Jira and Airtable, Linear and Notion, or any other heterogeneous tracker pair in design and cross-functional work.

Schema
This kind of loss happens when a data field in one system has no clear equivalent in the other. Most syncs quietly skip the field and the absence goes unremarked, so the receiving team has no way to know information was missing. Translation Engine names the field explicitly in the manifest so the gap is visible and the field can be wired up in a later pass or grammar revision.
Semantic
Names match across systems, but behavior doesn't. A "High" priority in Jira drives SLA timers, notifications, and project-level reporting; in Airtable the same word is just a label on a single-select field. Most syncs paper over the gap and a receiving team assumes parity. Translation Engine flags the mismatch and names the side effects that didn't make the trip, so anyone reading the manifest can see what behavior they're not getting.
Hierarchy
Parent–child and membership relationships flatten in transit. An epic link becomes a text label; a sprint object collapses to its name with dates and state stripped off. A receiving system that doesn't model these relationships absorbs the data anyway and the structure quietly disappears. Translation Engine records what structure existed in the source and what survived to the destination, so the gap is auditable rather than invisible.
Context
Sometimes the substantive spec lives outside the source system to begin with — a Slack thread, a Notion page, a Figma comment — and the link to it travels but the content behind the link does not. A naive sync moves the URL and the destination viewer assumes the ticket is self-contained. Translation Engine surfaces these embedded references explicitly, so a reader knows up front that the real content lives somewhere else and where to look.
Provenance
Destination rows can end up orphaned from their source — a Jira key dropped during translation, an Airtable record with no trail back. Most syncs treat the new record as the source of truth and break the audit chain. Translation Engine preserves a bidirectional mapping inside the manifest itself, so any destination row can always be traced back to its origin and the two systems stay reconcilable.

2Manifest — loss inventory

  • Semantic description→Description Jira stores descriptions as a structured document with formatting, mentions, and embedded actions. Translation flattens that to plain text — the structure does not survive the move.
  • Context Slack thread URL in description The URL is preserved in the destination, but the conversation it points to is not. A viewer in Airtable sees a link without the thread behind it and has no way to fetch the spec.
  • Semantic status→Status "Active" in Jira's workflow is bucketed to "In Progress" in Airtable. The workflow transitions, permissions, and allowed next states behind the source status have no destination equivalent.
  • Semantic priority→Priority The label "High" maps across, but the SLA timers, notification rules, and reporting behaviors that Jira priority triggers do not. The destination row carries the word without the behavior.
  • Hierarchy parent→Epic The typed link to SCRUM-2 is flattened to the epic's name as text. Navigation, roll-up tracking, and child-completion behavior in Jira do not have an equivalent in the destination.
  • Hierarchy sprint→Sprint The Jira sprint object — name, state, start and end dates, completion date — collapses to a single string in Airtable. Multi-sprint membership and the time window are dropped.
  • Hierarchy fixVersions→Fix Version The array of version objects in Jira is joined by commas into one text field. Release dates and release status do not survive.

Mapped 1:1: summary→Name, labels→Labels, key→Jira Key (preserved via the manifest's bidirectional ID mapping).

The pattern

Two composable layers carry the work, and the file-system shape matches one-to-one.

Per-platform adapters

Typed clients for each system — jira.ts, airtable.ts. Auth, HTTP plumbing, env access, and error formatting live in a shared lib/, so adding a new platform is one adapter file with zero new plumbing.

Per-context grammars

Opinionated translation rules for one work-context — project-management.ts here. Adding a new context (design system operations, critique rituals, roadmapping prioritization) is one grammar file. Each is deliberately opinionated, not configuration.

Every pass is a discrete snapshot, not a continuous sync. The manifest is the deliverable; the destination write is secondary. That order forces every translation decision to be named before it's executed.

What this isn't

  • Not a sync product. Each pass is a snapshot. No webhooks, no continuous mirroring. Webhooks are a later slice if they're needed at all.
  • Not a generic schema translator. Per-context grammars are opinionated by design. Adding a new context is a deliberate scoping pass, not configuration.
  • Not a hosted service. A TypeScript library plus this demo. Run it yourself, against your own systems, on your own terms.
